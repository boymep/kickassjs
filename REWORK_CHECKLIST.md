# Чеклист переработки секции

Используется при каждом следующем разделе после `binary-search`, `two-pointers`, `sliding-window`. Эталон — то, как они выглядят сейчас.

## Главный принцип текста

Писать так, как объясняет коллега, а не учебник или маркетолог. Простыми словами, по делу, без пафоса и преувеличений. Не «революционная техника», а «приём, который сводит O(n²) к O(n)». Не «алгоритм мечты», а «работает, когда массив отсортирован». Если предложение можно сократить без потери смысла — сократить.

## 1. Чистка legacy в `src/data/lessons/<topic>.ts`

- [ ] Удалить поле `resources` (videos + links)
- [ ] Удалить поле `flashcards`
- [ ] Удалить поле `interviewQA`
- [ ] Удалить импорты типов `Flashcard` и данных из `data/flashcards/`
- [ ] На уровне `Chapter` убрать: `video`, `links`, `flashcardIds`, `docsLink`
- [ ] Оставить только: `topicId`, `intro`, `chapters`, `finalQuiz`, `cheatsheet`, `nextTopics`

## 2. Intro

- [ ] `whyItMatters` — 1–2 коротких абзаца: что это, где применяется, зачем учить. Без лирики.
- [ ] Длина `whyItMatters` ≥ 80 символов (проверяется тестом)
- [ ] `interviewAngle` — одно нейтральное предложение про ключевой инсайт
- [ ] `prerequisites` — массив с уже пройденными темами (если есть)

## 3. Главы

- [ ] 5–6 глав, логически последовательных
- [ ] Структура главы: краткая теория → визуализация (если применимо) → код → callouts → checkpoint
- [ ] Обязательно есть глава «Подводные камни» (или эквивалент) — типичные ошибки
- [ ] В подходящих местах — `playground` с `expectedOutput`
- [ ] Callouts разных типов используются осмысленно: `info` / `tip` / `warning`

## 4. Checkpoints (обязательно интерактивные, с проверкой ответа)

- [ ] Каждая глава имеет хотя бы один checkpoint
- [ ] Используются разные типы: `ordering`, `match-pairs`, `multi-select` + вопросы из квиза
- [ ] ID всех checkpoint-вопросов из квиза добавлены в `CHECKPOINT_IDS`, чтобы не дублировались в `finalQuiz`
- [ ] Никаких «flashcards без проверки» — только то, где можно дать неверный ответ и получить фидбек

## 5. Cheatsheet

- [ ] Структура: когда применять → сложность → шаблоны кода → когда не работает → граничные случаи
- [ ] Нейтральный тон, без воды

## 6. nextTopics

- [ ] 1–2 темы с конкретным обоснованием «почему эта дальше»

## 7. Тон текста — ЧТО УБИРАТЬ

- [ ] Образные фразы: «отскакивать от зубов», «лакмусовая бумажка», «золотое объяснение», «коварный случай»
- [ ] Судейские формулировки: «junior / middle / senior», «слабый сигнал», «интервьюер на вашей стороне», «плюс-балл к внимательности», «сильно прибавляют в оценке»
- [ ] Жаргон: «в проде», «собес», «фейл», «бомж-вариант»
- [ ] Обращения второго лица: «вы должны», «вам дают», «вы знаете», «вы умеете», «вы сами», «вашу внимательность» → безличные обороты
- [ ] Фальшивые проценты: «в 80 % случаев», «9 из 10 кандидатов» → «часто», «обычно»
- [ ] Лозунги: «никто в здравом уме», «каждый знает», «все так делают»
- [ ] Глагол «должен / обязан» в адрес читателя

## 8. Тон текста — ЧТО ИСПОЛЬЗОВАТЬ

- [ ] Живой, человеческий язык — но без преувеличений и пафоса
- [ ] Безличные обороты: «применяется», «используется», «возвращается»
- [ ] Технические термины на русском, где есть устоявшийся перевод
- [ ] Названия классов JS (`Map`, `Set`, `Array.prototype.indexOf`) — оставлять как есть
- [ ] Названия задач LeetCode — переводить на русский; номер можно оставить в скобках («LeetCode 76»)
- [ ] Если предложение можно сократить без потери смысла — сократить
- [ ] Усилители «самый», «очень», «крайне», «абсолютно», «безусловно» — только когда без них не передать факт

## 9. Английские фрагменты в тексте — заменить

- [ ] `fixed-size` / `dynamic-size` → «фиксированное окно» / «динамическое окно»
- [ ] `sliding window` → «скользящее окно»
- [ ] `two-pointers` (в прозе, не в slug) → «два указателя»
- [ ] `prefix sums` → «префиксные суммы»
- [ ] `monotonic deque` → «монотонная очередь»
- [ ] `FAANG` → «крупные компании» / «топовые компании»
- [ ] `must-have`, `must-solve`, `must-know` → «обязательная задача», «нужно знать»
- [ ] Заголовки разделов глав — на русском (LeetCode-номер можно в скобках)

## 10. Визуализации

### Структура
- [ ] **Не** использовать `<Paper>` — это создаёт вложение в Chapter
- [ ] Обёртка: `<Box>` с тонкой рамкой (`border: 1, borderColor: 'divider'`) и subtle фоном
- [ ] **Не** дублировать заголовок главы внутри визуализации
- [ ] Кратко обозначить вход данных в одну строку

### Цвета
- [ ] **Никакого** хардкода `#FFFFFF`, `#F2F2F7`, `#005EC4` и т. п.
- [ ] Палитра собирается через `useTheme()` / `theme.palette.mode` в одном месте сверху компонента
- [ ] Проверить читаемость в тёмной теме (фон, текст, статусные блоки)

### Начальное состояние
- [ ] При `stepIdx === 0` — пустое состояние без подсветок
- [ ] Текущий шаг: `const step = stepIdx > 0 ? steps[stepIdx - 1] : null`
- [ ] При `step === null` — плашка «Нажмите Следующий шаг, чтобы начать»

### Указатели
- [ ] Под клетками — цветные label-пилюли (L/R/M/S/F и т. п.) **вместо** SVG-стрелок
- [ ] Цвет пилюли согласован с подсветкой клетки

### Контролы
- [ ] Кнопки: MUI `Button size="small"`, `variant="contained"` для «Следующий шаг», `variant="outlined"` для «Сбросить»
- [ ] Иконки: `ArrowForwardIcon` для шага, `RestartAltIcon` для сброса
- [ ] Кнопка «Сбросить» disabled, когда `stepIdx === 0`
- [ ] Кнопка «Следующий шаг» disabled, когда дошли до конца

### Статусная плашка
- [ ] Внизу одна плашка с описанием текущего шага
- [ ] Состояние «успех» — зелёный фон через theme, не хардкод
- [ ] Текст русский, без `sum`, `maxSum` и т. п. в подписях (оставлять только в коде)
